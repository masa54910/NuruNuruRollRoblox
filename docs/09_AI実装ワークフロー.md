# 09 AI実装ワークフロー

## 役割

| 担当 | 役割 |
| --- | --- |
| Work | 設計、調査、コードレビュー、工程管理 |
| Codex | ローカル実装、静的確認、Git操作、事実と未確認の整理 |
| GitHub Copilot | VS Code内での局所的な補助提案 |
| Roblox Studio | 実行環境、Script Analysis、Output、物理・Networkテスト |
| Mako | 見た目、操作感、実際の挙動の最終確認 |
| GitHub | コード、文書、復元ポイントの共有 |

AIはStudioで観測していない結果を「成功」「修正済み」「完成」と断定しません。

## 標準工程

1. 現状確認: Git、関連docs、コード、Studio/Rojo状態。
2. 仮説または設計: 原因か目的を一文にし、受入条件を決める。
3. 最小実装: 一工程、一つの主制御、一つの責任。
4. 静的確認: diff、検索、sourcemap、構文、対象外差分。
5. Studio同期: 正しいprojectとRojo接続先を確認。
6. MakoによるPlay確認: 操作感、見た目、再現条件。
7. 不具合報告: 期待、実際、Output、頻度、再現手順。
8. 最小修正: 新しい観測に対応する一変更。
9. 再確認: 元の症状と周辺機能の回帰。
10. 復元ポイント: 合格状態をcommit。
11. push: remote競合がない場合だけ通常push。
12. 次工程: docsの順序に従い、一工程だけ開始。

## AIへの依頼テンプレート

```text
目的:
確認済みの現状:
期待結果:
実際の結果:
変更してよいファイル:
変更禁止ファイル:
Studioでの受入条件:
Git保存条件:
```

## 原因調査の型

```text
観測 → 仮説 → 一つの比較テスト → 結果 → 採否
```

同じframeの値だけでなく、書込直後と次simulationを分けます。方向、所有権、別writer、Humanoid、衝突を一度に変えません。失敗した仮説も記録し、同じ調整を繰り返しません。

## 停止条件

- remoteとlocal historyが競合。
- 未コミット変更の所有者や意図が不明。
- Studio側にRojo管理外の競合Scriptがある。
- API仕様または現在の挙動を確認できない。
- 物理、見た目、操作感にMakoの判断が必要。
- 受入条件を満たすために対象外fileの変更が必要。

この場合は状態、影響、選択肢を報告し、勝手に範囲を広げません。
