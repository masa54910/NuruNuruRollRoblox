# 03 クライアントサーバー設計

## 責任分離

Clientは入力、カメラ、ローカル表示、低遅延の予測を担当します。Serverはゴール、スコア、報酬、生成物、共有状態、Remote入力の正当性を決めます。Clientが送った結果ではなく、可能ならClientの意図を受け取りServerが検証します。

## NetworkOwnership

Roblox物理はassemblyの計算をServerまたはClientへ分散します。所有Clientは滑らかな応答を得られますが、その物理結果をServerが完全には検証できません。

ぬるぬるRollでは、操作対象RootのownerをStudioで観測してから制御場所を決めます。`SetNetworkOwner` はServer APIであり、必要性を証明せず固定しません。Server所有にすると安全性は上がる一方、操作遅延やジッタの可能性があります。

## RemoteEvent / RemoteFunction

- Event: 一方向通知。返答を待たない処理向け。
- Function: 同期応答が必要な限定用途。yieldと失敗の影響が大きいため多用しない。
- Remote名と方向、payload形式、頻度上限を文書化する。
- Remoteを追加する前に、既存の `Remotes.lua` と配置を確認する。

## Server検証

| 検証 | 例 |
| --- | --- |
| 型 | `typeof(value) == "number"`、Instance class |
| 有限性 | NaN、`math.huge` を拒否 |
| 範囲 | steer -1..1、距離、速度上限 |
| 頻度 | player単位rate limit |
| 権限 | 対象が本人のCharacterか |
| 文脈 | ラウンド中、接地中、Goal付近 |
| 空間 | Server観測位置との距離 |

不正入力はまず無視・clamp・正しい状態へ補正し、証拠なしに強い処罰をしません。

## 物理ゲームの方針

- Client予測を採用しても、ゴールとタイムはServerが再確認。
- `Touched` だけで報酬を確定しない。対象、player、位置、cooldownを検証。
- 毎フレームRemote送信を避け、入力変化または低頻度snapshotを使う。
- 位置・速度の許容差は坂、ジャンプ、network latencyを考慮する。
- owner切替、respawn、複数playerで再現する。
