# 06 デバッグとテスト

## 原則

観測、仮説、最小変更、再観測の順で進めます。数値を上げるだけ、複数方式の同時置換、成功ログだけでの完了判定を避けます。

## Studioの確認面

- Script Analysis: 静的なLuauエラーと警告。
- Output: 実行時error、warn、print。ServerとClientの文脈を分ける。
- Developer Console: 実行中のClient/Serverログ、memory、network。
- Physics / Network owner visualization: ownershipやassemblyの診断。
- Explorer / Properties: 実際の親階層、属性、Part property。

## テストモード

| Mode | 用途 | 注意 |
| --- | --- | --- |
| Play | Spawnからの通常体験 | ServerとClientが動く |
| Play Here | 任意地点の局所確認 | Spawn問題の確認には不向き |
| Run | CharacterなしのServer挙動 | Map生成やServer初期化 |
| Server & Clients | replication、Remote、複数player | 1～2 clientsから開始 |

## 標準診断手順

1. 再現条件、期待、実際、頻度を一文ずつ記録。
2. `git status` と対象コードを確認。
3. 同じ値を書くScriptを全文検索。
4. 初期化時、書込直後、次simulation、一定時間後を分けて観測。
5. 一つの仮説だけを切り替える。
6. 成功条件と失敗条件を同じログに出す。
7. 修正後に元の症状と周辺機能を回帰確認。
8. 一時ProbeとDebug markerを停止・撤去する。

## 物理Probe

- Root position / velocity / assembly root / mass / anchored。
- Humanoid state / floor material / WalkSpeed / PlatformStand / AutoRotate。
- Network ownerはServer側から取得。
- 方向vector、magnitude、Road軸とのdot。
- 前方Raycastのhitとdistance。
- 書込前、直後、next heartbeat、0.1秒後、0.5秒後。

ログはprefixを固定し、1秒に1回程度へ制限します。一度押しProbeは通常controllerを止めて競合を排除します。

## Raycast可視化

一時的なNeon PartやBeamはDebug folderに置き、`Anchored=true`、`CanCollide=false`、`CanTouch=false`、`CanQuery=false` とします。本番フラグでは生成せず、巨大な赤球は復活させません。

## 人間が確認する項目

- コースがStartからGoalまで視覚的に連続しているか。
- 入力の向き、速度感、滑り、ジャンプ、酔い。
- Characterが意図せず跳ねる、回る、浮く、壁へ張り付くか。
- 再起動やRojo再接続後も同じか。
- mobile/gamepadで操作可能か。

これらをCodexがStudioで観測できない場合は「未確認」と報告します。
