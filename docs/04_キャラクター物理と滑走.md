# 04 キャラクター物理と滑走

## 目標と前提

ぬるぬるRollはVehicleSeatを使わず、Roblox Characterの `HumanoidRootPart` assemblyを物理主体として滑らせる構想です。ただし現在は制御方式の採用試験中で、完成した滑走物理はありません。まず一つの主制御だけを選び、Studioで速度が入るか、残るか、地形に沿うかを確認します。

## Character物理の構成

- `Humanoid`: 歩行、ジャンプ、状態遷移、標準制御に関与。
- `HumanoidRootPart`: Character assemblyの代表Root。速度、質量、ownershipの観測点。
- `AssemblyLinearVelocity`: assemblyの現在線速度。直接設定は瞬時の速度変更。
- `AssemblyAngularVelocity`: assemblyの角速度。暴走診断と回転減衰に使用可能。
- Humanoid標準挙動と独自物理が同時に速度を変更すると競合する。

## 制御手段の比較

| 手段 | 性質 | ぬるぬるRollでの候補 | 注意 |
| --- | --- | --- | --- |
| 重力＋低摩擦 | Engineの自然力 | 基本となる第一候補 | 平地では減速設計が必要 |
| `ApplyImpulse` | 一回の瞬間的な運動量変化 | ジャンプ、衝突、診断用 | 毎フレーム使用しない。質量に依存 |
| `VectorForce` | 一定の力を継続付与 | 補助推進、重力方向補正の候補 | 制限なしでは速度が増え続ける |
| `LinearVelocity` | 目標速度を維持するConstraint | 制御速度が必要な場合の候補 | 自然な重力滑走を打ち消し得る |
| `AlignOrientation` | torqueで姿勢を目標へ合わせる | 回転安定が必要な後工程の候補 | 強すぎると地形・ジャンプと競合 |
| `AssemblyLinearVelocity` 直接設定 | 即時に速度を変更 | 一度押しProbe、限定的な補正 | 毎フレーム全成分上書きは自然力を消す |
| CFrame / Tween | 位置を運動学的に変更 | プレイヤー移動では不採用 | 衝突、重力、replicationと整合しにくい |

古いBodyMover系を新規採用せず、必要ならAttachmentベースのMover Constraintsを比較します。

## 速度上書きと力の加算

`AssemblyLinearVelocity = ...` はその瞬間の速度を置換します。重力、接触、Humanoid、別Scriptが次のsimulationで変えるため、代入直後の値だけでは成功判定できません。一方、Forceは質量と時間に応じて速度を変えます。

毎フレーム速度を上書きすると、坂による自然加速、上りの失速、壁衝突、着地反動を消しやすくなります。採用する場合も、必要な成分だけを限定し、書込箇所を一つにします。

## 地面と坂方向

Raycastで得る情報は `Position`、`Normal`、`Instance` です。Rayのdirectionには長さを含めます。Character自身をfilterで除外し、Course上のRoadかを検証します。

重力 `g` を床面へ投影した接線方向は概念的に次です。

```lua
local tangentGravity = gravity - normal * gravity:Dot(normal)
```

実際にはRoadの生成順forwardと照合し、下り方向が逆転しないことを確認します。Normalだけではカーブの進行方向は決まりません。

## 接地・平地・上り・空中

- 接地: Raycast hit、距離、Road判定、法線、vertical velocityを組み合わせる。
- 平地: 接線重力が小さい。自然な摩擦で減速させ、最低速度保証は別工程で判断。
- 上り: 進行速度を増やさず、重力により失速させる。
- 空中: Ground Stickを切り、重力と限定的なair steeringだけにする。
- 着地: 接地復帰時の法線速度を観測し、演出と物理補正を分離する。
- 壁: 床法線と壁法線を区別し、横速度・前速度の保持率を後工程で調整する。

## 摩擦と物性

`CustomPhysicalProperties` のDensity、Friction、Elasticityとweightは、接触相手との組合せで結果が変わります。床だけを極端な低摩擦にして完了とせず、Character接触、壁、Lotion、CourseをStudioで比較します。跳ねる場合はElasticity、衝突形状、速度、複数BodyPartの接触も確認します。

## NetworkOwnership

RootがClient所有なら入力応答は良くなりますが、Serverはその物理結果を信頼できません。Root ownerはServer ProbeとStudioのNetwork owner可視化で確認します。ownerを明示設定するかは、Client一度押しとServer一度押しの差を測ってから決めます。

## 採用判断の手順

1. 速度・Force・CFrame・Humanoid stateを書き込む全箇所を列挙。
2. Character LookVectorへ一度だけ速度を与え、直後・次frame・0.1秒・0.5秒を観測。
3. 必要ならServer側でも同じProbeを実施しownership差を確認。
4. Humanoid条件を一項目ずつ比較。
5. 重力のみで坂を下る最小ケースを確認。
6. 平地減速、上り失速、左右入力を別工程で追加。

## 不採用・禁止

- プレイヤー移動へのCFrame、PivotToの連続使用、Tween。
- VehicleSeat。
- `LinearVelocity` と速度直接上書きを同時に主制御にすること。
- 強いDownForceを常時加えること。
- Studio未確認の物理を完成扱いにすること。
