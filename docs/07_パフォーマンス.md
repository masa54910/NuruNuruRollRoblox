# 07 パフォーマンス

## 方針

計測前の大規模最適化は行いません。長大コースと大量Partがあるため、frame callback、Raycast、Workspace探索、物理assembly数を早期から観測可能にします。

## frameイベント

- `RenderStepped`: Localのcameraや描画直前処理。Server logicに使わない。
- `Heartbeat`: simulation後の更新。physics観測や一部controller候補。
- `Stepped` / simulation events: physics timingが必要な限定用途。
- event駆動で済む処理を毎frame pollingしない。
- `dt` を使用し、frame rate依存の加速・減衰を避ける。

## 高コストになりやすい箇所

- 毎frameの `Workspace:GetDescendants()` や名前検索。
- 毎frameのtable/Instance/RaycastParams生成。
- Characterごとに重複した接続。
- 数千Partすべてへの毎frame処理。
- Debug printの大量出力。
- 複数Scriptによる同一assembly制御。

Roadは初期化時に生成順cacheを作り、近傍indexだけを調べます。RaycastParamsも対象folderが安定している間は再利用を検討します。ただしcache invalidationを複雑化する前に測定します。

## Partと物理

- Anchored Course Partは動かさない。
- 不要なCanTouch / CanQuery / collisionを用途別に外す。
- decorative Partを物理simulationへ参加させない。
- 接続されたunanchored Partとassembly数を確認する。
- 大量生成は一度だけ行い、重複MapをHealthCheckで検出する。

## StreamingEnabled

約78,000 studs規模の想定では将来候補ですが、導入すると遠方RoadのClient参照、raycast、cache、goal表示に影響します。現状で有効とは確認していません。Phase 2の基本物理完成前に推測導入しません。

## 計測

- Performance SummaryでCPU/GPU/memory/networkの傾向を見る。
- MicroProfilerでframe spikeと継続負荷を分ける。
- `debug.profilebegin/end` は重い区間を限定して測る。
- ServerとClientを別々にprofileする。
- mobile相当deviceで最終確認する。
