# 05 入力とカメラ

## 入力の責任

入力取得はClientで行います。Phase 2の基本操作は自動的な坂滑走に対する左右ライン取りで、前進キーによる人工加速は初期工程に含めません。Keyboardだけで設計を固定せず、Actionを中心にtouchとgamepadへ拡張できる形を目指します。

## UserInputServiceとContextActionService

- `UserInputService`: device状態や生のInputBegan / Changed / Endedを観測する。
- `ContextActionService`: 名前付きactionを複数deviceへbindし、標準操作をsink/passする。
- UI入力を奪わないよう `gameProcessedEvent` を確認する。
- press/releaseを対で管理し、focus喪失やCharacter再生成時にstateをresetする。

現在の `DownhillController.client.lua` はA/D、左右矢印、gamepad Xを直接扱います。これは存在確認済みですが、Phase 2の最終input設計として採用済みではありません。

## 標準Character Controlsとの競合

標準Control Module、`Humanoid:Move`、WalkSpeed、独自速度制御が同時に働くと、前進や横速度を打ち消す可能性があります。disableする場合は、失敗時・CharacterRemoving時・機能停止時に必ずenableへ戻します。歩行停止、jump停止、physics適用を一度に変更せず比較します。

## 左右入力

- 入力値を -1..1 に正規化。
- deadzoneを設ける。
- frame rateに依存しないよう `dt` で加速度を計算。
- Road forwardに直交する接線rightへ作用させる。
- 入力解除時は横速度を急に0へせず、設計した減衰を適用。
- 前進方向の回転と横速度付与は別の効果として計測する。

## ジャンプ

ジャンプはPhase 2の工程6です。標準JumpRequestをそのまま使うか、接地時だけImpulseを許可するかは未決定です。多重発火、空中連打、Client申告だけの接地、質量差を考慮します。

## カメラ

- `CameraType.Custom`: 標準Character camera。
- `CameraType.Scriptable`: LocalScriptがCamera CFrameを管理。
- `CameraSubject`: Customへ戻す際にCharacterのHumanoid等を再設定。
- プレイヤー移動へのCFrameは禁止ですが、カメラ表示のCFrame制御とは別物。
- `RenderStepped` は描画直前に必要なcamera更新へ限定し、physics主制御には使わない。

現在の `DownhillCamera.client.lua` はConfigで停止中です。Scriptable camera、速度FOV、roll、collision raycastのコードは存在しますが、Studioで採用確認済みではありません。

## POVと酔い対策

- まず安定した滑走を完成させてから有効化。
- positionとrotationの応答を分ける。
- roll、shake、FOVの上限を低く始める。
- 着地shakeは短時間で減衰し、常時振動させない。
- camera collisionで壁内部を避ける。
- respawn時にlocal transparency、FOV、CameraTypeを復元する。

## モバイル・ゲームパッド

Keyboard検証後、ContextActionServiceまたはInput Action方式で左右actionを抽象化します。Studio Device EmulatorとController Emulatorでtouch button、thumbstick deadzone、UI占有を確認します。
